## 0. 이 개정판의 핵심 (TL;DR)

1. **표본 평탄화 → 결론 불변.** IT_기타 69.9% 가중을 제거(직군 단순평균)해도 4개 지표가 ±1.4pp 내로 일치. 직군 간 분산이 작아 가중 편중은 *경험적으로 무의미*.
2. **선호(HL2 6:4)는 약한 앵커 → 행동지표로 전환.** 의사결정을 HL2 선호(60:40)가 아니라 HL1(불안)·HL4(재클릭)·**HL5(대기 유발 이탈 Δ)** 같은 *행동* 지표 위에 세운다.
3. **HL5 신규 측정 = 결정타.** "기다린 뒤 실패"는 "즉시 실패"보다 이탈을 **+17.6pp** 더 유발. 대기는 측정 가능한 순수 비용이다.
4. **대기 → 재클릭 피드백 루프 확인.** 비관적 대기가 HL4 재클릭(≈59%)을 유발 → 큐 적체 → 대기 악화. 차단기는 **멱등성 키**뿐.
5. **결론(고확신, 6:4에 안 기댐): `fail-fast 사전체크` + `멱등성 키`가 결정적. 락 종류는 2차.**

## 2. 방법론 보정 — 표본 평탄화

5-3에서 제기된 "IT_기타(69.9%)가 가중평균을 지배한다"는 우려를 검증하기 위해, **직군별 단순평균(equal-weight)**으로 재집계해 n-가중평균과 비교한다.

| 지표 | n-가중평균 | 평탄화(단순평균) | Δ | 해석 |
|------|-----------|----------------|---|------|
| HL1 ≤3초 불안 | 74.8% | **74.8%** | ~0 | 직군 무관 균일 |
| HL2 B(확정 선호) | 58.6% | **60.0%** | +1.4 | 평탄화해도 여전히 ~6:4 |
| HL3 이탈 | 41.6% | **41.3%** | −0.3 | 사실상 동일 |
| HL4 재클릭 | 59.6% | **58.9%** | −0.7 | 사실상 동일 |
| HL5 이탈 Δ(대기유발) | 18.0pp | **17.5pp** | −0.5 | 사실상 동일 |

> **결론:** 직군 간 분산이 ±8pp로 작아 **가중/평탄 결과가 일치**한다. 표본 편중 우려는 *원리적으론 타당하나 이 데이터에선 결론을 바꾸지 못한다.* 단, **세그먼트별 전략**(예: 프론트/모바일 대상 fail-fast UI)을 세울 때는 평탄·가중 평균이 아닌 *해당 세그먼트 수치*로 판단한다. 이하 본문은 **평탄화 기준**으로 서술한다.

## 4. 가설별 전체 현황

### HL1 — 결제 대기 인내

```
1초 안에 불안       ████████████ 28%
2~3초까지 기다림    ███████████████████ 47%   ← 누적 ≤3초 = 74.8%
5초까지 기다림      ███████ 18%
10초 이상도 기다림  ███ 7%
```
**판정: ✅ 74.8%가 3초 내 불안 (3:1, 결정적).** → 락/트랜잭션 타임아웃 **2~3초 상한**, 정상 응답 1초 이내 목표.

### HL2 — 경합 실패 방식 선호 (즉시 vs 확정)

```
A 즉시 품절(fail-fast)  ████████████████ 40.0%
B 확정 결과(기다려도)   ████████████████████████ 60.0%
```
**판정: ⚠️ B 우세이나 60:40 (약한 앵커).** 선호 자체로는 락을 결정하기에 근거가 약하다 → HL5로 대체 측정.

### HL3 — 결제 실패 후 행동 (매출)

```
같은 사이트 재시도   ████████████████ 41%
나중에 다시 방문     ███████ 17%
다른 사이트(경쟁사)  ████████████ 30%   ┐
구매 포기            █████ 12%          ┘ 이탈 = 41.3%
```
**판정: ✅ 이탈 41.3%.** → **false-fail(헛 실패)은 곧 매출 누수.** 재고가 실제 있는데 동시성 충돌로 실패시키면 이 41%가 손실로 전환.

### HL4 — 느린 응답 시 중복 제출 (멱등성)

```
거의 항상 재클릭   ████████ 16%
가끔 재클릭        ██████████████████ 43%   ← ①+② = 58.9%
거의 안 함         ███████████ 29%
절대 안 함         █████ 12%
```
**판정: ✅ 58.9% 재클릭.** → 락 종류와 무관하게 **멱등성 키 필수**.

### HL5 — 실패 방식별 재이용 의향 (대기의 순수 비용) ⭐⭐

| 상황 | '망설여진다'+'안 쓸 것 같다' (이탈 위험) |
|------|------------------------------------------|
| **A. 즉시 실패** | 34.2% |
| **B. 기다린 뒤 실패(3초)** | **51.7%** |
| **Δ (대기가 유발한 추가 이탈)** | **+17.5pp** |

```
A 즉시 실패        █████████████████ 34.2%
B 대기 후 실패     ██████████████████████████ 51.7%
                   └──────── Δ +17.5pp = 대기의 순수 비용 ────────┘
```
**판정: ✅ Δ +17.5pp (≥10pp 기준 크게 초과, 전 직군 양수).** → **"대기 후 실패"는 "즉시 실패"보다 이탈을 17.5pp 더 유발.** 같은 사람이 두 시나리오를 겪었을 때의 *행동 델타*이므로 HL2 선호(6:4)보다 훨씬 결정적이고 표본 가중에 둔감하다.

## 5. 대기 → 재클릭 피드백 루프 (HL1 × HL4)

비관적 락의 "대기"는 단독 비용이 아니라 **HL4 재클릭을 유발해 스스로를 악화시킨다.**

```
비관적 락 대기(2~3초, 고경합 시)
  → HL1: 74.8%가 3초 내 불안
  → HL4: 58.9%가 재클릭
  → (멱등성 없으면) 재클릭 = 새 요청 = 같은 락 큐에 재진입
  → 큐 길이↑ → 대기↑ → 더 불안 → 더 재클릭   ← 양의 피드백(retry storm)
  → DB 커넥션 풀 고갈 위험
```

- **"대기 시 악화 인구"**(3초 내 불안 **AND** 재클릭) ≈ **45~52%** (독립가정 44% / 양의 상관 보정 ~52%).
  - ⚠️ *추정치*: 결합분포를 직접 묻는 크로스탭은 미수집. 두 지표가 동일 직군에서 함께 높아(프론트·모바일) 양의 상관을 가정.
- **차단기는 멱등성 키 하나뿐.** 재클릭이 같은 요청 키로 dedup되면 새 큐 대기자가 생기지 않아 루프가 끊긴다.
- ❗ **단, 이 루프는 "2~3초 대기"가 실제로 발생할 때만 작동한다.** 임계구역이 짧은(순수 DB) 평상시엔 비관적 대기가 <100ms라 HL1·HL4가 트리거되지 않는다. 루프는 **고경합 꼬리(hot-item) 상황 전용 리스크**다 — 그리고 그게 바로 락을 도입하는 이유인 상황이다.

## 6. 락 결정 권고 ★ (재집필 — 행동지표 기반)

### 6-1. 엔지니어링 사실 (설문과 독립)

- 임계구역 = `재고 조회 → reduce → 주문 저장 → 커밋` (순수 DB, 외부 호출 없음) → holder 점유 **수~수십 ms**
- 경합 = "인기 상품 마지막 1개에 다수 집중" → **고경합 · 짧은 임계구역**

### 6-2. 무게중심 이동: "비관 vs 낙관"이 아니라 "대기 제거 + 중복 차단"

선호(HL2 6:4)는 약한 앵커였다. 행동지표로 다시 세우면 **6:4에 기대지 않는 고확신 결론** 두 개가 남는다.

| 결론 | 확신도 | 근거 |
|------|--------|------|
| **fail-fast 사전체크 = 필수** | 🔴 높음 | HL1(75% 3초 내 불안) + HL5(대기 후 실패가 +17.5pp 더 이탈). 명백 품절은 락 큐에 넣지 말고 *즉시* 반환 → 경합 시 *대중*을 대기 없이 처리 |
| **멱등성 키 = 필수** | 🔴 높음 | HL4(59% 재클릭)의 피드백 루프 차단기 |
| 락 종류(비관 vs 낙관) | 🟡 2차 | 사전체크가 대중을 흡수하면 락엔 *진짜 동시 진입한 소수*만 남음 → 대기 수십ms → UX 차이 소멸 |

### 6-3. 왜 락 종류가 2차가 되는가 (경합 시나리오 분해)

마지막 1개에 100명이 몰릴 때:
- **순수 비관적**: 100명 전원이 행 락에 줄섬 → 1명 성공, 99명이 *차례로 대기 후* 품절 확인 → 대기 꼬리 + 재클릭 유발 (HL5의 +17.5pp 비용 발생)
- **순수 낙관적**: 전원 즉시 충돌 → 99명 *즉시* 품절(fail-fast) → 대기 없음 (HL5 비용 회피), 단 재시도 로직 필요
- **사전체크 + 락(종류 무관)**: 1번째 커밋 후 들어온 요청은 사전체크에서 quantity=0 확인 → *즉시* 품절(대기 없음). 실제 락에 진입하는 건 커밋 전파 전 *극소수 동시 진입자*뿐 → 이들에 한해 짧은 락 중재

→ **사전체크가 낙관적의 fail-fast UX를 흡수**한다. 그 뒤 소수 경합만 락이 중재하므로, 짧은 임계구역에선 비관적(단순·결정적)이든 낙관적(재시도)이든 사실상 동등. **권고: 사전체크를 깐 뒤 비관적 락을 1차로 (구현·테스트 단순, false-fail 없음).** 낙관적은 임계구역이 길어지면(외부 결제 연동 등) 재검토.

### 6-4. 종합 설계 (우선순위)

```
1. [필수·고확신] fail-fast 사전체크 — 명백 품절은 대기 없이 즉시 반환
2. [필수·고확신] 멱등성 키 — 재클릭 중복주문 차단 (락과 별개 레이어)
3. [필수] 락/트랜잭션 타임아웃 2~3초 — 무한 대기 차단
4. [선택·2차] 락 종류 — 비관적 1차(사전체크 후 소수 경합만 중재). 임계구역 길어지면 낙관적 재검토
```

> **보고/과제 서술 논리 순서:** ① 물리(짧은 임계구역·고경합) → ② 행동지표(대기 후 실패가 +17.5pp 더 이탈) → ③ 그래서 "대기 제거(사전체크) + 중복 차단(멱등성)"이 핵심 → ④ 잔여 소수 경합은 비관적 락으로 단순·확정 처리. 이 순서가 6:4 선호에 기대지 않아 가장 단단하다.

## 부록: 적용 지점 (방향 — 구현 아님)

```
OrderFacade.placeOrder()  @Transactional (타임아웃 2~3초)
  ├─ (사전·비락) 재고 빠른 조회 → 명백 품절이면 즉시 실패 (대기 없음)   ← HL1·HL5 대응
  ├─ StockRepository.findForUpdate(productId)   ← 비관적 락 진입점(소수 경합만)
  ├─ StockModel.reduce(quantity)                ← read-modify-write 안전
  └─ 주문 생성·저장 → 커밋 시 flush + 락 해제
멱등성: 요청 ID 기반 중복 제출 차단 (락과 별개 레이어)            ← HL4 대응
```
