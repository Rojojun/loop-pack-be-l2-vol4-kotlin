# 유저 설문조사 — Run 03 (주문/결제 대기·실패 경험)

> **목적:** 주문 동시성 제어(락) 선택을 *유저 불편·매출 영향* 근거로 정당화한다.
> **대상:** IT 직군 기술자 (Run 02와 동일 패널, 15개 유효 직군)
> **소요 시간:** 2분 이하 / 4문항
> **페르소나 출처:** nvidia/Nemotron-Personas-Korea 데이터셋
> **연결 결정:** 재고 차감 동시성 — 비관적 락(queue) vs 낙관적 락(fail-fast)

---

## 왜 이 설문인가 (그리고 함정)

락 선택의 **지배 변수는 유저가 아니라 물리** — (1) 경합률, (2) 임계구역 길이다.
우리 케이스의 임계구역은 순수 DB(외부 호출 없음, 포인트/결제 없음)라 수~수십 ms로 짧고,
경합은 "마지막 1개에 몰림"이 핵심이다. **이 조건에서 두 락은 평상시 둘 다 1초 안에 응답**하므로,
"몇 초까지 기다리나요?"만 물으면 **두 락이 구분되지 않는다.**

두 락이 갈리는 지점은 단 하나 — **마지막 1개를 두고 경합할 때 '진 사람'의 경험**:

| 락 | 진 사람의 경험 |
|----|---------------|
| 비관적 | 줄 서서 기다린 뒤 "품절" (확정 결과, 대기 있음) |
| 낙관적 | 즉시 "품절" (fail-fast, 대기 없음) |

따라서 이 설문은 "최대 대기 시간"이 아니라 **① 경합 실패를 어떤 방식으로 겪고 싶은가,
② 실패하면 실제로 무엇을 하는가(=매출)** 를 측정한다.

---

## 기획자 검토 원칙 (Run 02 계승)

```
✓ 기능/기술 이름을 먼저 언급하지 않는다 (락·트랜잭션 용어 금지, confirmation bias 방지)
✓ "~하면 쓸 것 같나요?"가 아닌 "지금 어떻게 하고 있나요?"로 묻는다
✓ 기술 용어보다 일상 언어로 작성한다
✓ 각 문항은 하나의 가설만 검증한다
✓ 단, 두 UX 모델 비교는 강제 트레이드오프(A/B)로 제시한다 (yes-bias 회피)
```

---

## 검증 가설

| ID | 가설 | 락 결정에 주는 신호 |
|----|------|---------------------|
| **HL1** | 결제 응답이 늦으면 N초 만에 불안해진다 | 락/트랜잭션 **타임아웃 상한** (락 공통) |
| **HL2** | 경합 실패는 "즉시 품절"보다 "기다려도 확정 결과"가 낫다 | **비관 vs 낙관 직접 판별** ⭐ |
| **HL3** | 결제 실패/지연 시 경쟁사 이탈 또는 포기한다 | **매출 손실 정량화** ⭐ |
| **HL4** | 응답이 느리면 결제 버튼을 다시 누른다 | 멱등성/중복주문 방어 필요성 |
| **HL5** | "기다린 뒤 실패"는 "즉시 실패"보다 더 큰 이탈을 유발한다 | **대기의 순수 비용(이탈 Δ) 직접 측정** ⭐⭐ |

---

## Q1. 결제 대기 인내 (HL1 검증)

**온라인에서 '구매하기'나 '결제'를 눌렀는데 화면이 바로 안 바뀔 때, 보통 몇 초쯤 지나면 "뭔가 잘못됐나?" 하고 불안해지나요?**

```
○ 1초 안에 결과가 안 나오면 바로 불안하다
○ 2~3초까지는 기다린다
○ 5초 정도까지는 기다린다
○ 10초 이상도 기다린다
```

> **검증:** '1초 안'+'2~3초' 합산(≤3초 불안) 비율 → 락/트랜잭션 타임아웃 상한 결정
> **왜 이렇게 물어보는가:** "얼마까지 기다릴 수 있나요?"(의향)가 아니라 실제 불안이 시작되는 시점(경험)을 묻는다.

---

## Q2. 경합 실패 방식 선호 (HL2 검증) ⭐

**인기 상품의 마지막 재고를 여러 명이 동시에 사려는 상황입니다. 둘 중 어느 쪽이 덜 답답한가요?**

```
○ A: 누르자마자 바로 "품절"이라고 알려준다 (기다림 없음)
○ B: 2~3초 기다리더라도 "구매 성공 / 품절"의 확실한 결과를 받는다
```

> **검증:** A 우세 → **낙관적 fail-fast** 정당화 / B 우세 → **비관적 확정 결과** 정당화
> **왜 이렇게 물어보는가:** 락 용어 없이 두 UX 모델을 강제 트레이드오프로 직접 비교한다. "기다리는 게 괜찮나요?"는 yes-bias를 유발하므로 A와 B를 동시에 제시한다.

---

## Q3. 결제 실패 후 행동 (HL3 검증) ⭐

**사고 싶던 물건을 결제하려다 실패했거나 너무 오래 걸린 경험이 있나요? 그때 어떻게 했나요?**

```
○ 같은 사이트에서 다시 시도했다
○ 잠시 후 / 나중에 다시 들어왔다
○ 다른 사이트(경쟁사)에서 샀다
○ 그냥 안 샀다 (구매 포기)
```

> **검증:** '경쟁사 구매' + '포기' 합산 = **직접 매출 손실률**
> **왜 이렇게 물어보는가:** 가정이 아니라 실제 행동 경험을 물어 매출 영향을 정량화한다.

---

## Q4. 느린 응답 시 중복 제출 (HL4 검증)

**결제 버튼을 눌렀는데 반응이 느릴 때, 버튼을 한 번 더 눌러본 적 있나요?**

```
① 거의 항상 다시 누른다
② 가끔 누른다
③ 거의 안 누른다
④ 절대 안 누른다
```

> **검증:** ①+② 비율 → 중복주문 방어(멱등성 키) 필요성 (락 종류와 무관하게 필수)
> **왜 이렇게 물어보는가:** 느린 응답이 만드는 실제 행동을 물어 멱등성 설계 근거를 확보한다.

---

## Q5. 실패 방식별 재이용 의향 (HL5 검증) ⭐⭐ — 가장 결정적

**다음 두 상황을 모두 겪었다고 상상해보세요. 각각 이 가게를 다음에 다시 이용할 가능성은?**

**상황 A — 즉시 실패:** 마지막 1개를 사려고 눌렀더니 *누르자마자* "품절"이 떴다.

```
○ 다시 이용한다   ○ 망설여진다   ○ 다시 안 쓸 것 같다
```

**상황 B — 기다린 뒤 실패:** 마지막 1개를 사려고 눌렀더니 *3초간 "처리 중"이 돌다가* "품절"이 떴다.

```
○ 다시 이용한다   ○ 망설여진다   ○ 다시 안 쓸 것 같다
```

> **검증:** (B의 '망설여진다'+'안 쓸 것 같다') − (A의 동일 합산) = **'대기'가 추가로 유발하는 이탈률 Δ.** 이 Δ가 비관적 락 대기의 *순수 비용*이다.
> **왜 이렇게 물어보는가:** HL2가 '선호'(6:4, 약한 앵커)를 물었다면, Q5는 **같은 사람**에게 두 시나리오를 모두 겪게 해 *행동 변화의 차이(동일인 내 델타)*를 직접 잰다. 집단 간 선호 비율이 아니라 개인 내 델타라 훨씬 결정적이고, 표본 가중 방식에도 둔감하다.

## 결과 → 락 결정 매핑 (해석 기준표)

| 가설 | 측정 문항 | 우세 결과 | 락 결정 시사점 |
|------|---------|---------|----------------|
| HL1 | Q1 ≤3초 불안 비율 | ≥ 70% | 타임아웃을 2~3초 안쪽으로 (락 공통) |
| HL2 | Q2 A/B | A 우세 | 낙관적 fail-fast 정당화 |
| HL2 | Q2 A/B | B 우세 | 비관적 확정 결과 정당화 |
| HL3 | Q3 이탈(타사+포기) | ≥ 40% | false-fail이 매출 직격 → 확정성(비관) 또는 견고한 재시도(낙관) 필수 |
| HL4 | Q4 ①+② | ≥ 50% | 락과 독립적으로 **멱등성 키 필수** |
| HL5 | Q5 (B이탈 − A이탈) Δ | ≥ 10pp | 대기가 이탈을 유의하게 유발 → **대기 최소화(fail-fast 사전체크) 필수** |

## 한계 (반드시 명시)

이 설문은 락을 **"결정"하는 게 아니라 "정당화"하는 보조 증거**다.
최종 판단 = `설문(UX·매출 증거)` + `물리(경합률·임계구역 길이)`.
둘이 충돌하면 **물리가 우선**한다. 예: HL2가 "즉시 품절(낙관)"로 나와도, 임계구역이 짧아
비관적 대기가 ~1초면 낙관적 재시도 복잡도를 감수할 이유가 약해진다.
설문 결과는 "감점 없는 선택지를 고르는 근거"로 사용한다.
