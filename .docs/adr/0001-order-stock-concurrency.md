## 1. Context (배경)

### 문제
- `OrderFacade.placeOrder()`는 `@Transactional`이나 **락이 없다.** `StockModel.reduce()`는 전형적인 read-modify-write(`quantity -= amount`)라, 마지막 1개 재고에 동시 주문이 들어오면 **Lost Update → 초과판매**가 발생한다.
- 과제 Must-have: 트랜잭션 + Lock + 동시성 테스트.

### 처음의 오해
당초 이 결정을 **"비관적 락 vs 낙관적 락 택1"** 이라는 단순 선택으로 봤다. 그러나 분석 과정에서 더 근본적인 사실이 드러났다.

### 핵심 통찰 ⭐
> **사용자의 불만(빡침)을 결정하는 것은 *락의 종류*가 아니라, 비즈니스 플로우에서 *실패가 발생하는 타이밍*이다.**
> - 빠른 실패(투자/대기 전) → 덜 불만
> - 늦은 실패(기다린 뒤 / 결제 진행 후) → 더 큰 불만

Run 03 가상 설문이 이를 수치화했다 (⚠️ 시뮬레이션, 보조 근거):
- **HL5**: "기다린 뒤 실패"는 "즉시 실패"보다 이탈을 **+17.5pp** 더 유발 (전 직군 양수).
- **HL1**: 74.8%가 결제 응답 1초 내 불안 → 응답 예산 ~**1000ms**.
- **HL4**: 58.9%가 느린 응답 시 재클릭 → 대기는 재클릭을 유발하는 피드백 루프.
- **HL2**(선호 60:40)는 약한 앵커라 의사결정 근거에서 제외.

### 물리적 사실 (설문과 독립)
- 임계구역 = `재고 조회 → reduce → 주문 저장 → 커밋`. **순수 DB, 외부 호출 없음**(포인트·결제 단계 미포함) → holder 점유 수~수십 ms로 짧다.
- 따라서 **우리의 짧은 플로우에서는 비관/낙관 간 UX 차이가 작다.** "낙관=결제까지 갔다 회수당하는 분노"는 *긴 체크아웃*에서 최악이며, 우리 케이스엔 해당 구간이 없다.

## 2. Decision Drivers (결정 기준)

1. **정합성**: 초과판매 0건 — 협상 불가 (필수)
2. **빠른 실패**: 늦은 실패가 +17.5pp 더 이탈(HL5), 1초 내 불안(HL1)
3. **중복주문 방어**: 59% 재클릭(HL4)
4. **구현·테스트 단순성**: 과제 채점에서 "동시 2건 → 1성공/1실패" 증명이 결정적
5. **짧은 임계구역**: 비관적 대기 비용이 작다는 물리적 전제

---

## 3. Considered Options (검토한 선택지)

| 옵션 | 정합성 | UX(빠른 실패) | 구현 복잡도 | 판정 |
|------|:---:|:---:|:---:|------|
| A. 락 없음 (현状) | ❌ 초과판매 | — | 낮음 | **기각** |
| B. 낙관적 락 (`@Version`+재시도) | ✅ | △ (재시도 지연/폭주) | 높음(재시도·테스트) | 보류 |
| C. **비관적 락 (`SELECT … FOR UPDATE`)** | ✅ | ✅ (사전체크 결합 시) | 낮음(확정·단순) | **채택** |
| D. 조건부 UPDATE (`quantity >= :q`) | ✅ | ✅ | 낮음(단 도메인 로직 SQL 이전) | 대안 |

> B(낙관적)는 짧은 플로우에선 정합성·속도 모두 무방하나, 고경합(마지막 1개)에서 재시도가 폭주하고 "충돌→재시도→최종 실패" 흐름의 테스트가 까다롭다. C가 같은 정합성을 더 단순·확정적으로 제공한다.
> D(조건부 UPDATE)는 락 없이 원자적 차감 + 빠른 실패를 동시에 달성하는 강력한 대안. 단 `reduce()`의 도메인 규칙을 SQL `WHERE`로 내리는 트레이드오프가 있어, 과제의 "Lock" 학습 목표와는 거리가 있다. **C의 백업/대안으로 기록.**

## 4. Decision (결정)

> **비관적 락(PESSIMISTIC_WRITE)을 1차 채택한다.**
> 그 위에 **Fail-Fast 다층**과 **멱등성 키**를 더해, "정합성은 락이, UX는 빠른 실패가" 책임지도록 분리한다.

선택 이유는 *"낙관이 빡치게 해서"가 아니라*:
1. 우리 플로우가 짧아 비관/낙관 UX 차이가 작고,
2. 비관적이 false-fail 없이 **확정적**이며 구현·테스트가 단순하고,
3. **Fail-Fast 사전체크**가 대기를 1000ms 예산 안으로 눌러주기 때문이다.

---

## 4-1. 락 방식 상세 결정 (Lock Mechanics)

락 구현 방식을 다음 세 가지로 확정한다.

### (1) 락 모드 — `PESSIMISTIC_WRITE` (배타락), READ 아님
재고 차감은 `SELECT → reduce → UPDATE`의 read-modify-write이며, SELECT는 *쓰기를 위한 읽기*다. **락 모드는 SQL 동사(SELECT)가 아니라 의도(쓰기)로 정한다.**
- `PESSIMISTIC_READ`(공유락)는 여러 트랜잭션이 동시에 같은 행을 읽게 허용 → 둘 다 마지막 1개를 읽고 UPDATE 시 X락으로 **승격 충돌(데드락)** 또는 초과판매.
- `PESSIMISTIC_WRITE`(배타락)는 한 트랜잭션만 보유 → 읽는 순간 배타락을 선점해 승격 자체를 없앰 → 깔끔한 직렬화.

### (2) 대기 정책 — 블로킹(기본), NOWAIT·양수 타임아웃 지양
- `jakarta.persistence.lock.timeout` **양수 값(예 3000)은 MySQL에서 무시**된다. MySQL `FOR UPDATE`엔 per-query 양수 타임아웃 문법이 없고, Hibernate가 emit 가능한 건 NOWAIT(`0`)/SKIP LOCKED(`-2`)뿐. 양수는 그냥 기본 블로킹(`innodb_lock_wait_timeout`, 기본 50s)으로 떨어진다.
- **NOWAIT도 지양.** NOWAIT은 "경합=패배"로 단정해, 재고가 충분한데도 동시 접근만으로 헛 실패시킨다(재고 100개여도 동시 2건 중 1건 실패). 대부분의 주문은 '마지막 1개'가 아니므로 흔한 경우를 망친다.
- **결정: 힌트 없는 블로킹 `FOR UPDATE`.** 재고 있는 경합은 수 ms 대기 후 모두 성공. fail-fast는 락이 아니라 **L1 사전체크**가 담당(품절만 빠르게 거름).
- 대기 상한이 필요하면 per-query가 아니라 세션 레벨 `innodb_lock_wait_timeout`(예 3~5s)으로. 단 커넥션 전역 부수효과 인지. L1 + 짧은 임계구역이면 실제 대기가 ms 단위라 우선 **기본값으로 시작**.

### (3) 구현 메커니즘 — `@Lock` + 파생 쿼리 (`@Query`/QueryDSL 불필요)
정적 lock-by-id이므로 Spring Data 파생 쿼리 + `@Lock`이면 충분하다.
```kotlin
@Lock(LockModeType.PESSIMISTIC_WRITE)
fun findWithLockByProductId(productId: Long): StockModel?   // FOR UPDATE (블로킹)
```
- ⚠️ 파생 쿼리는 락 의미어를 **`By` 앞**에 둔다(`findWithLockBy…`). `findByProductIdForUpdate`처럼 `By` 뒤에 `ForUpdate`를 붙이면 속성 경로로 해석돼 깨지므로, 그 이름을 쓰려면 `@Query` 필요(+`:param` 이름 일치).
- QueryDSL `.setLockMode(PESSIMISTIC_WRITE)`는 **동적/복잡 락 쿼리**용. 정적 단일 조회엔 보일러플레이트만 늘어 채택하지 않는다.
- 일반 조회와 **차감용 락 조회를 별도 메서드로 분리**한다 (같은 트랜잭션 1차 캐시로 락 SQL이 스킵되지 않도록 L1 사전체크는 projection/비영속 조회 권장).

## 5. Fail-Fast 다층 설계 — "가장 이른 계층에서 실패시킨다" ★

빡침의 원인이 *늦은 실패*이므로, 실패를 **최대한 앞 계층에서, 투자/대기 전에** 발생시킨다. 뒤로 갈수록 비싸다.

| 계층 | 기법 | 무엇을 빨리 거르나 | 비용/비고 |
|------|------|------------------|-----------|
| **L0 트랜잭션 진입 전** | 입력 검증 (수량>0, 동일 productId 병합, 존재 확인) | 잘못된 요청 | 락·트랜잭션 안 잡고 즉시 400 |
| **L0** | 멱등성 키 선조회 | 재클릭 중복 제출 | 처리중/완료면 기존 결과 즉시 반환 → 큐 재진입 차단 (HL4) |
| **L0** | (선택) 매진 플래그 캐시(Redis) | 핫 아이템 매진 | 최적화일 뿐 — **진실은 DB**, 정합성 근거로 쓰지 않음 |
| **L1 트랜잭션 내·락 전** | 재고 사전체크 (non-locking read) | "명백 품절"인 대중 | 경합 시 99명을 락 큐에 안 세움. **TOCTOU 레이스는 L2가 닫음** |
| **L2 락 획득·차감** | 블로킹 `FOR UPDATE` (PESSIMISTIC_WRITE, 힌트 없음) | — (fail-fast 아님, **정합성** 담당) | 재고 있는 경합은 수 ms 대기 후 성공. NOWAIT 지양(헛 실패). 상세 §4-1 |
| **L3 차감** | (대안) 조건부 UPDATE `… WHERE quantity >= :q`, affected=0 → 실패 | 마지막 1개 경합 패자 | C(락)와 **택1** 관계. 락 채택 시엔 도메인 `reduce()` 사용 |
| **횡단** | 트랜잭션/락 구간 최소화 (외부호출·무거운 조회 배제) | 대기 자체 | 점유 시간 = 동시성 한계 |
| **횡단(FE)** | 클릭 즉시 버튼 disable + "처리 중" 표시 | 재클릭 *유발* 억제 | HL4 근본 완화 (서버 멱등성과 이중 방어) |

### 핵심 조합 (우리가 가는 길)
```
L0 입력검증 + 멱등성 선조회
   └ 통과한 요청만 트랜잭션 진입
L1 재고 사전체크(비락) → 명백 품절이면 즉시 실패 (대기 0, 락 큐 제외)
L2 SELECT … FOR UPDATE (블로킹, 힌트 없음) → 진짜 동시 진입 소수만 직렬화 (재고 있으면 잠깐 대기 후 성공)
L3 도메인 reduce() → 안전한 read-modify-write
   └ 커밋 시 flush + 락 해제
```
- **L1이 "빠른 실패"를, L2가 "정합성"을 담당** — 역할이 깔끔히 분리된다.
- L1만으로는 안전하지 않다(check-then-act 레이스). 그 레이스를 닫는 것이 정확히 L2 락이다. **사전체크는 락을 대체하지 않는다.**

## 6. Consequences (결과)

### 긍정
- 초과판매 0 (정합성).
- 매진은 즉시 통보(L1), 정상 차감도 짧은 임계구역이라 평상시 1000ms 한참 안쪽.
- 재클릭 중복주문 차단(멱등성).
- 구현·동시성 테스트 단순.

### 부정 / 리스크
- **락 보유 시간 = 동시성 한계** → 락 구간(L2~커밋)에 외부 호출·대량 조회를 절대 넣지 않는다.
- 고경합 시 일부 대기(블로킹) → 상한이 필요하면 세션 `innodb_lock_wait_timeout`(3~5s)으로. **NOWAIT·양수 `lock.timeout` 힌트는 지양**(MySQL이 무시하거나 헛 실패 유발). 상세 §4-1.
- 조건부 UPDATE(D) 대비 락 경합 트래픽이 더 생길 수 있음 (단 사전체크가 대중을 걸러 완화).

### 재검토 트리거 (이 결정을 뒤집을 조건)
- 임계구역에 **외부 결제(PG) 연동**이 추가되어 holder 점유가 수백 ms~초로 길어질 때 → 비관적 락 위험 → **낙관적 락 / 예약(reservation) / 사가**로 재검토.
- 경합이 실측상 드물다고 확인될 때 → 낙관적이 처리량 유리.

## 7. Validation (검증 — 동시성 테스트)

- **재현 우선**: 락 없는 상태에서 `재고=1`, 동시 2건(`ExecutorService`+`CountDownLatch`) → "둘 다 성공/재고 음수"가 깨지는 것을 먼저 확인.
- **방지 검증**: 락 적용 후 동일 테스트 → "성공 1 / 실패 1 / 최종 재고 0".
- **함정**: ① 스레드별로 *각자의 트랜잭션*에서 호출(한 트랜잭션 안에서 돌면 재현 안 됨). ② H2는 락 동작이 실DB와 달라 가짜 통과 위험 → 동시성 테스트는 **실 MySQL**(docker-compose 제공) 권장.

---

## 8. 한 줄 요약

> 정합성은 **비관적 락**이, 사용자 만족은 **빠른 실패(사전체크)**가, 중복은 **멱등성**이 책임진다. 락은 핵심 *도구*이되, UX의 결정자는 *실패 타이밍*이다.

## 부록: 적용 지점 (방향 — 구현은 직접)

```
OrderFacade.placeOrder()  @Transactional(timeout = 2~3s)
  ├─ [L0] 입력 검증 + 멱등성 키 선조회 (트랜잭션 밖이 이상적)
  ├─ [L1] StockRepository.findByProductIdIn(...)  // 비락 사전체크 → 명백 품절 즉시 실패
  ├─ [L2] StockRepository.findWithLockByProductId(productId) // 비관적 락 (블로킹 FOR UPDATE)
  ├─ [L3] StockModel.reduce(quantity)
  └─ 주문 생성·저장 → 커밋(flush + 락 해제)
```
> `findWithLockByProductId`는 `@Lock(PESSIMISTIC_WRITE)` + **파생 쿼리**(JPQL/QueryDSL·힌트 불필요). 일반 조회와 **차감용 락 조회를 메서드로 분리**한다. 상세 §4-1.
> `@Transactional(timeout)`은 *트랜잭션* 타임아웃(≠ 락 타임아웃)이며 그대로 유효.
